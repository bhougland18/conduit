## Recent Work

`conduit-core` now has a tiered metadata policy adapter:

- `MetadataTier` names control, data, and high-cost data metadata classes.
- `TieredMetadataPolicy::control_only()` records control facts and drops data-tier records by default.
- `TieredMetadataPolicy::sample_data_every(...)` samples data-tier facts with a deterministic counter.
- `TieredMetadataPolicy::with_high_cost_data()` explicitly opts in to high-cost data records.
- `TieredMetadataSink<S>` wraps any existing `MetadataSink` without changing the `MetadataSink` trait.
- `TieredMetadataSink::record(...)` treats current `MetadataRecord` values as control facts because they do not carry payload bodies.
- `TieredMetadataSink::record_with_tier(...)` lets future payload-shape or Arrow metadata call sites apply data/high-cost policy explicitly.
- Tests cover default control forwarding, default data/high-cost dropping, deterministic sampling, and explicit high-cost opt-in.

`conduit-cli` now has first-pass `run` and `explain` commands:

- `conduit explain <workflow.json>` validates the workflow and prints topology, scaffold execution mode, and metadata policy details.
- `conduit run <workflow.json> <metadata.jsonl>` validates the workflow, runs it through the scaffold no-op executor, and writes lifecycle metadata JSONL.
- CLI `run` uses `AsupersyncRuntime`, `conduit-engine::run_workflow_with_metadata_sink`, `JsonlMetadataSink`, and `TieredMetadataSink`.
- `run_workflow_with_metadata_sink` was added to `conduit-engine` so workflow-level execution can emit runtime metadata without changing `NodeExecutor`.
- The CLI still does not load real node implementations; `run` is currently a scaffold execution path that proves validated topology, runtime scheduling, and metadata capture.
- Tests cover `explain` output, JSONL metadata capture, and engine lifecycle metadata recording.
